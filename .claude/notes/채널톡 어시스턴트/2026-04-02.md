# 2026-04-02 — cs-agent-demo

## 11:30 — 유저 메시지만으로는 매니저 답변 유형 예측 불가 발견

채널톡 환불 문의 389건 데이터로 agent 답변 생성 파이프라인을 구축하면서, 템플릿 매칭 eval을 돌려봄.

**읽은 파일**: `채널톡 답변 유형 분석.md`, `채널톡 모호한 문의 패턴 분석.md`, `CS AI 에이전트 데모 설계.md`, `src/admin_api.py`, `src/refund_agent.py`, `data/refund_test_cases.json`

**의사결정**:
- 처음 접근: LLM이 유저 메시지 → 의도 분류 → 자유 생성. 결과가 매니저 답변과 완전히 다름
- 두번째 접근: 실제 매니저 340건 응답에서 템플릿 7종 추출 → LLM은 분류 + 변수 채우기만. 훨씬 나음
- 발견: 유저 메시지만으로는 템플릿 매칭 정확도 13.2%. **매니저는 유저 정보를 조회한 후 답변 유형을 결정**함

**수치**:
- 환불 테스트 케이스: 389건 (BQ channel_io.messages + chats에서 추출, userId 100% 매핑)
- 매니저 응답 있는 건: 340건
- Ground truth 분포: T99_기타 61.2%, T1_해지안내 11.5%, T2_환불규정 8.8%, T3_접수완료 8.5%, T4_자동결제설명 6.5%
- mock 분류기 정확도: 45/340 (13.2%) — T99_기타 제외하면 더 높을 수 있지만, T99가 최다라 의미 없음
- 주요 오분류: `T1→T99` 174건 (해지 키워드지만 매니저는 비정형 답변)

**코드 발견**:
- `src/templates.py` → 매니저 응답에서 추출한 실제 템플릿 7종 정의
- `src/refund_agent_v2.py` → 3단계 파이프라인: 분류 → 도구 호출 → 변수 채우기
- `scripts/eval_template_matching.py` → ground truth 기반 매칭 정확도 측정

**배운 것**:
- agentic AI에서 핵심은 "유저 메시지 → 답변"이 아니라 "유저 메시지 → **정보 조회** → 조회 결과 기반 의사결정 → 답변"
- 유저 메시지 = 의도 힌트, 실제 답변 분기 = 시스템 데이터에 의존
- 싱글턴(해지 안내처럼 바로 답변)과 멀티턴(추가 질문 후 답변)을 분리해야 함
- "해지요청" → 조회해보니 이미 열람함 → 환불 규정 안내. 이 분기는 유저 메시지만으로 예측 불가

## 14:15 — 워크플로우 그래프 + eval loop + 시점 복원

옵시디언 다이어그램 → `src/workflow.py` 코드화. eval loop 9회 반복. 시점 기반 데이터 복원.

**읽은 파일**: `src/refund_agent_v2.py`, `src/workflow.py`, `scripts/eval_loop.py`, `data/refund_test_cases_enriched.json`

**의사결정**:
- LLM 프롬프트 분류 → 워크플로우 그래프(룰 기반)로 전환. 다이어그램 그대로 코드화
- 대화 맥락 로직 추가 → 정확도 하락(14.2%→10.1%). 과도한 규칙. 보수적 재적용으로 복구

**챌린지**:
- 과거 데이터의 현재 API 상태 불일치. 해결: 거래 이력(불변)에서 대화 시점 기준 환불 전/후 판단
- 대화 맥락 규칙이 양날의 검. "이전 T2 → T3" 규칙이 조건 없이 적용되면 오히려 악화

**수치**:
- eval 9회: 0% → 11.7% → 13.9% → **14.2%**(최고) → 10.1%(맥락 과적용) → 복구 → **14.2%**(시점 복원)
- T99 제외: 45/128 (35.2%). T1 79%, T3 21%, T2 13%, T4 14%
- 389건 중 343건 실제 admin API 데이터 enriched (40초, 에러 0)

**코드 발견**:
- `workflow.py:run_workflow()` → 다이어그램 노드를 순차 분기로. `WorkflowContext`에 데이터 누적
- `eval_loop.py --suggest` → 오분류 패턴별 개선 제안 + `--history`로 정확도 추이

**배운 것**:
- eval loop는 강력하지만 데이터 품질(시점 복원)이 선행돼야 의미 있음
- 규칙 추가 → 회귀 체크 필수. 한 패턴 고치면 다른 패턴 깨짐
- T99(60%)가 천장. 정형 템플릿끼리(T99 제외) 비교가 실질적 지표
- 다음: 키워드 mock → LLM/임베딩 유사도로 의도 분류 개선

## 15:00 — 근본 문제 발견: 골든셋 비일관성 + 의도 분류 한계

임베딩 유사도, 데이터 기반 분기, mock 데이터 스냅샷 등 여러 접근을 시도했지만 정확도가 안 오르는 근본 원인을 발견.

**읽은 파일**: `src/workflow.py`, `tests/test_workflow.py`, `data/mock_test_cases.json`

**사고 흐름**:
1. 임베딩 유사도로 의도 분류 → T99 제외 42.2% (키워드 35.2%에서 개선)
2. "의도 분류 자체가 병목" → 데이터만으로 분기하자 → 22%로 하락
3. 의도만으로도 안 되고 데이터만으로도 안 되는 이유: **같은 데이터 상태에서 매니저마다 다른 템플릿 사용**
4. 결론: 골든셋(매니저 응답) 자체가 비일관 → 어떤 알고리즘도 정확도 한계

**수치**:
- 키워드 의도분류 + 워크플로우: 14.2% (T99 제외 35.2%)
- 임베딩 유사도 + 워크플로우: 17.1% (T99 제외 42.2%)
- 데이터만 분기 (의도 없이): 22% — T1이 3%로 폭락
- mock 데이터 스냅샷 + 의도분류: 33.1% — 시점 복원보다 나음
- 워크플로우 유닛 테스트: 13/13 (100%) — **로직 자체는 맞음**

**챌린지**:
- T1(해지방법), T2(환불규정), T4(자동결제설명)가 **데이터 상태가 동일** (active 구독 + 미환불 결제). 차이는 매니저 재량
- "해지요청" + 미환불 결제 → 매니저 A는 T1, 매니저 B는 T2. 데이터로 구분 불가, 의도로도 구분 불가
- 한 쪽 규칙 고치면 반대 쪽 깨짐 (eval loop에서 확인: +2% 후 -4% 반복)

**배운 것**:
- 매니저 답변을 "따라하기"가 아니라 **일관된 정책을 정해서 적용**하는 게 맞음
- 골든셋 대비 정확도가 아니라 **정책 준수율**이 올바른 평가 지표
- 다음 스텝: CS팀과 정책 확정 ("이 상황엔 이렇게 답한다" 통일) → 그 정책대로 워크플로우 구현 → 정책 기반 eval
- 기존 패턴 중 하나를 정책으로 선택하면 "틀릴 수는 없음" — 이미 매니저가 쓰던 답변이니까

## 15:30 — 상담사 답변 비일관성 확인 + 정책 결정

mock 데이터 127건의 데이터 상태별 매니저 답변을 펼쳐서 비교. 같은 상황에서 매니저마다 다른 패턴 사용하는 것을 수치로 확인.

**읽은 파일**: `data/mock_test_cases.json`

**의사결정**:
- 비일관 분석: 127건 중 117건(92%)이 같은 데이터 상태에서 다른 템플릿 적용
- 가장 심한 그룹: "결제2건_환불0건" 40건에서 T1/T2/T3/T4/T8/T9 **6종**이 혼재
- 결론: 골든셋 대비 정확도는 의미 없음. **정책을 정해서 일관 적용**이 올바른 접근

**핵심 발견 — 같은 상황에서 매니저마다 다른 답변**:
- 유저 "해지요청" + 미환불 결제 있음 → 매니저 A는 T1(해지 방법만), 매니저 B는 T2(환불 규정까지)
- 유저 "환불" + 최근 결제 → 매니저 A는 T2(규정 안내), 매니저 B는 T4(리텐션 먼저)
- 이건 매니저 재량이지 시스템 규칙이 아님

**수치**:
- 비일관 케이스: 117/127건 (92%)
- 일관 케이스: 10/127건 (8%) — 대부분 1건짜리 고유 상태
- 최대 충돌 그룹: "결제2건+이전매니저O" 40건 → 6개 템플릿 혼재

**정책 결정 (제안)**:
| 데이터 상태 | 선택 | 이유 |
|---|---|---|
| 결제 없음 | T1 (해지 방법) | 환불할 게 없음 |
| 미환불 결제 있음 | T2 (환불 규정+금액) | 정보가 더 많아 유용 |
| 전부 환불됨 | T3 (접수 완료) | 이미 처리됨 |
| 최근건만 환불됨 | T1 (해지 방법) | 환불 끝, 해지만 남음 |
| 이전 턴에 T2 있음 | T3 (접수 완료) | 규정 안내 → 처리 |
| T4(리텐션) | 빼기 | 매니저 재량이지 정책 아님 |

**배운 것**:
- CS 자동화에서 "매니저 따라하기"는 함정. 매니저 간 비일관성이 높으면 따라할 대상이 없음
- **정책 기반 접근**: "이 상황엔 이 답변"을 정하고 일관 적용 → eval도 정책 준수율로
- 기존 매니저 답변 중 하나를 고르면 "틀릴 수 없음" — CS팀과 확정하면 됨
- 데이터 분석 → 비일관 펼치기 → 정책 선택 → 워크플로우 구현 → 정책 기반 eval 순서가 맞았음

## 16:00 — 정책 확정 + 케이스 맵 완성

환불/해지 전체 케이스를 유저 메시지 패턴 × 데이터 상태로 펼치고, 확정 정책을 결정. T2(환불 규정) 안에서 전액/부분 환불 분기도 명시.

**읽은 파일**: `data/mock_test_cases.json`, `환불 케이스 전체 맵.md`, `환불 답변 정책 비교 및 확정안.md`

**의사결정**:
- 환불 규정 안내(T2) 안에서 추가 분기 필요: 7일 이내+미열람 → 전액, 그 외 → 부분(금액 계산)
- 에셋은 멤버십 구독과 다른 상품 유형 → 일단 제외
- 상담사가 "환불 건"이라고 판단 후 명령어 치는 구조 → 문의 유형 자동 분류는 다음 단계
- LLM에 "골라서 답변 만들어"가 아니라 **확정된 플로우를 코드로** 구현 → 테스트 가능

**확정 플로우**:
```
유저 식별 불가 → 본인확인 요청
카드 키워드 → 카드변경 안내
결제 없음 → 해지 방법 안내
전부 환불됨 → 접수 완료
미환불 + 7일 이내 + 미열람 → 환불 규정 (전액) + 금액
미환불 + 7일 이내 + 열람 → 환불 규정 (부분) + 금액 계산
미환불 + 7일 경과 → 환불 규정 (부분) + 금액 계산
```

**코드 발견**:
- `src/workflow.py` — 정책 기반 분기로 전면 교체. 의도 분류 의존 제거
- `tests/test_workflow.py` — 14/14 통과 (정책 기반)
- 이전 턴 맥락: `last_user_ts` 기준으로 매니저 이전 턴만 참조해야 함

**배운 것**:
- "유저 메시지는 카드 관련인지만 판단" — 나머지 분기는 전부 데이터 상태로 결정
- 확정된 플로우가 있으면 LLM 판단에 의존하지 않아도 됨 → 결정론적 테스트 가능
- 다음: 이 정책대로 워크플로우 최종 구현 + 실제 API 연동 + 환불 금액 계산 검증
