# 2026-04-06 — cs-agent-demo

## 00:06 — 모호 유형 재질문 대응 — 실데이터 조사 + 보류 결정

환불 도메인 이외에도 유저가 첫 메시지를 짧게 쓰고 나가버리는 "모호 문의" 에 에이전트가 재질문할 수 있어야 한다는 Gayoon 요구. 실데이터 조사 3회 반복 후 ROI 낮다고 판단해 구현 보류.

**읽은 파일**: `data/test_cases/refund_convos_jan.json`, `data/test_cases/refund_test_cases_enriched.json`, `src/bigquery/channel_queries.py`, `src/bigquery/client.py`

### 타깃 정의가 3번 바뀐 과정

1차 타깃 — "봇 버튼 텍스트 단독 입력" (예: "구독 상품변경/결제정보 확인", "사이트 및 동영상 오류"). 내가 과거 세션 데이터에서 발견한 E 카테고리 9건 중 7건이 봇 label 이라 여기에 focus 함.

2차 타깃 — Gayoon 정정: "봇 버튼 누른 건 유저가 메뉴로 해소하고 나간 걸 수도 있어. 진짜 타깃은 **자유 텍스트로 '환불신청', '가입변경' 같이 짧게 쓰고 나가는 경우**." 타깃 완전 뒤집힘.

3차 타깃 — 다시 Gayoon: "봇 버튼 눌렀을 때도 상담사가 대응하는 패턴이 있을 수도 있으니 그것도 조사해줘." 봇 버튼 이탈 + 매니저 대응 케이스도 조사 대상으로 복귀.

**의사결정**:
- Gate 별도 모듈 `src/gate.py` 만들었다가 폐기 (LLM classify 2회 중복 호출 피하려고). wrapper 내부 intent 재사용 패턴으로 교체.
- Wrapper 반영 구현은 보류. 카탈로그 스크립트만 작성해 재실행 가능하게 보관.

**수치**:
- 골든셋 기반 초기 카탈로그 (과거 데이터): 489건 중 34건 (7%) 모호 후보, 그중 D 환불외 오픈 = **0건**
- BQ 1달 (2026-03-06~04-05) 전체 도메인 추출: **2225 대화**
- 필터 통과 (자유 텍스트 ≤15자, 환불 키워드 배제 없음): **17건 (0.8%)**
- 17건 중 수동 검토로 "진짜 재질문 필요" = **약 7건 (0.3%)**
- 환불 외 진짜 도메인 샘플: 선물하기 문의, 라이브 입장 안돼요, 다운로드창 안보여요, 정규수업 들으려면요, 안되네요 — 4~5건만

**챌린지**:
- `refund_convos_jan.json` 과 `refund_test_cases_enriched.json` 이 BQ 에서 환불/해지 필터링된 추출물이라 환불 외 도메인 샘플이 애초에 0건. "환불 외 모호" 질문에 답할 데이터 부족.
- BQ 에서 환불 필터 제거하고 1달 재추출했지만 2225 대화 중 대부분은 여전히 환불 관련 (채널톡 CS 트래픽 자체가 환불 비중이 압도적).
- 매니저 "재질문 키워드" 로 필터링하면 T2 환불 규정 설명 같은 케이스 ("남겨주셔야" 문구 포함) 가 오탐됨. 매니저 응답을 필터 기준으로 쓰기 어려움.

### 대안 아이디어 — 채널톡 도큐먼트 KB + RAG (Gayoon 제안)

모호 문의에 규칙/LLM 분류로 대응하는 대신 **채널톡 도큐먼트 자체를 Bedrock Knowledge Base 에 넣고 RAG 로 긁어와서 답변** 하는 방향.

장점:
- 환불 도메인 카탈로그 작성 노력 최소화 (도큐먼트가 ground truth)
- 환불 외 도메인 (수업/배송/계정/기술) 도 도큐먼트만 있으면 커버 가능
- AgentCore KB 쇼케이스도 겸함 (해커톤 4/8 KB 항목 미이행 상태 해소)

고려사항:
- 도큐먼트 소스가 있는지 확인 필요 (CS팀 매크로 답변 모음, FAQ 문서 등)
- 도큐먼트 없으면 매니저 실답변 클러스터링으로 seed 생성 가능성 (demo-app-plan.md Part B 의 "템플릿 drift 감지" 와 맞물림)
- RAG 답변의 trust boundary: LLM 이 자유 생성 → intent_classifier 에 있던 "정책 = 코드가, 답변 = 코드가" 원칙과 상충 가능. KB 는 **참고용** 으로 쓰고 최종 답변은 템플릿 유지하는 하이브리드가 안전할 수 있음.
- 4/8 해커톤 이후 본격 착수.

**배운 것**:
- **실데이터를 먼저 까보면 타깃 정의가 흔들린다**. 1차 타깃 (봇 버튼) 이 2차 (자유 텍스트) 로 뒤집히고, 데이터 뽑아보니 진짜 빈도가 0.3% 라는 걸 알게 됨. 상상으로 구현했으면 엉뚱한 케이스에 비용 썼을 것. **데이터 탐색이 설계에 선행** 되어야 함.
- **BQ 필터 설정 자체가 데이터셋 편향의 원인**. "환불/해지" 키워드로 추출한 refund_convos_jan.json 이 환불 외 도메인 분석을 구조적으로 불가능하게 만듦. 필터 기준은 도구가 아니라 **데이터 정책**.
- **매니저 응답을 필터 기준으로 쓸 때 오탐 주의**. T2 환불 규정 안내 문구에 "남겨주셔야" 같은 일반 단어가 포함돼 재질문 키워드로 오매치. 매니저 응답은 결과론적 라벨링이지 입력 기준이 아님.
- **ROI 낮은 feature 는 구현 말고 기록으로 남기는 게 맞는 판단**. 0.3% 트래픽에 intent 이중 체계 정리 + @tool 추가 + SYSTEM_PROMPT 재설계 비용은 과투자. 해커톤 마감 있는 상황에선 특히.
