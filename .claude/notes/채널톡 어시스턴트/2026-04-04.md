**의사결정 — 데이터 구조 3층 분리**:

기존 문제: data/ 안에 16개 파일이 뒤섞여 있음 — 테스트 데이터, mock, eval 결과, 원본이 구분 안 됨. 파일 찾기 어렵고, 뭐가 재생성 가능하고 뭐가 원본인지 모호.

3층 구조로 분리:
```
data/
├── test_cases/       ← 유형별 테스트 데이터 (BQ에서 재생성 가능)
├── mock_scenarios/   ← 워크플로우 분기별 mock (수동 관리)
├── eval_results/     ← eval 결과 (스크립트로 재생성)
└── raw/              ← 변경 없는 원본
```

왜 이 구조가 필요한가:
1. **재생성 가능 여부가 명확** — raw는 건드리지 않고, test_cases/eval_results는 스크립트로 재생성
2. **역할별 분리** — mock은 수동 관리(정책 반영), test_cases는 BQ에서 자동 추출
3. **경로 참조 일관성** — 모든 코드에서 `data/{layer}/{file}` 패턴. 코드 리뷰 시 어떤 데이터인지 바로 파악
4. **파이프라인 방향이 보임** — BQ → test_cases → eval_results 순서

**의사결정 — "모호한 문의" 재정의**:

→ 분류: 문의 유형 → 각 유형에서 "바로 답변 가능?" 판단 → 불가능하면 추가 질문

## 15:30 — 모호한 문의 설계 결정: 유형이 아닌 상태로 편입

상담사 니즈 분석에서 나온 "모호한 문의 유형"을 별도 유형으로 분리하지 않기로 결정. 각 문의 유형 내에서 "모호한 상태"로 처리.

**읽은 파일**: `채널톡 모호한 문의 패턴 분석.md`

**의사결정 — 왜 별도 유형으로 안 나누는가**:

배경: 상담사들의 원래 요청은 "처음 메시지가 모호한 경우 추가 대응이 필요하다"였음. 그래서 처음에는 "모호한 문의"를 환불/해지, 로그인 등과 동등한 하나의 유형으로 분류하려 했음.

문제: 실제 데이터를 보니 모호함은 **모든 유형에서 발생**함:
- 환불인데 모호: "환불요망" (어떤 상품?)
- 로그인인데 모호: "안돼요" (뭐가 안 돼?)
- 맥락 없음: "네 이번달까지만요" (이전 대화 연속)

결정: **모호함 = 유형이 아닌 상태**. 매 메시지마다:
```
메시지 수신 → 문의 유형 분류 (환불, 로그인, 기술 등)
    ↓
해당 유형의 워크플로우 실행
    ↓
정보 충분? → Yes: 답변 생성
            → No: 추가 질문 생성 (A~D 패턴)
                A. 본인확인 (성함/전화번호)
                B. 상품특정 (어떤 과정?)
                C. 증상구체화 (어떤 화면?)
                D. 오픈질문 (무엇을 도와드릴까요?)
```

이점:
1. 분류기가 단순해짐 — "환불 vs 모호 vs 기타" 3분류가 아닌 "환불 vs 로그인 vs 기술" 유형만
2. 모호 대응이 모든 유형에 자동 적용 — 유형별 워크플로우 안에서 정보 충분성 체크
3. 멀티턴 자연스러움 — 추가 질문 → 유저 답변 → 같은 워크플로우 이어가기

**배운 것**:
- 상담사 니즈("모호한 문의 대응")와 시스템 설계("유형 분류")는 다른 층위. 니즈를 그대로 유형으로 만들면 안 됨
- 정보 충분성 판단은 **워크플로우 내부**에서 하는 게 자연스러움. 외부에서 선분류하면 오히려 복잡해짐

## 17:00 — Evaluation 설계: normal vs edge 케이스 + 실제 데이터 기반

환불/해지 100건을 LLM으로 분석해 워크플로우 커버 가능 여부 판단. 86% normal, 12% edge. edge 케이스의 실제 대화를 보고 evaluation 전략 수립.

**읽은 파일**: `data/test_cases/refund_convos_jan.json`, `data/test_cases/refund_pattern_analysis.json`

**수치**:
- 환불/해지 100건 중: 일반(normal) 43건 (86%), 튀는(edge) 6건 (12%), 에러 1건
- edge 6건 중 에이전트 대응 가능: 4건. 권한 필요(상담사): 2건
- LLM 분석 50건 92초 (1.8초/건)

**의사결정 — Evaluation을 두 가지로 나눔**:

문제: normal은 정답이 명확(템플릿+금액)하지만, edge는 정답이 하나가 아님. 매니저마다 다르게 대응.

해결: **normal = rule-based, edge = LLM-as-judge**

**normal 평가 (rule-based)**:
```
입력: 유저 "환불해주세요" + mock_api(미환불 500,000원)
기대: template=T2, 환불금액=360,000원
평가: 템플릿 일치? 금액 정확? → PASS/FAIL
```

**edge 평가 (LLM-as-judge)**:
실제 매니저 답변에서 **필수 요소**를 추출하고, 에이전트 답변에 그 요소가 있는지 체크.
```
실제 매니저: "콘텐츠 열람하셔서 전액환불 어렵습니다. ■ 환불 금액: 420,000원"
  → 필수 요소:
    ✅ 열람 여부 언급했는가?
    ✅ 전액 불가 안내했는가?
    ✅ 환불 금액 제시했는가? (420,000원)
    ✅ 규정 근거 설명했는가?

에이전트 답변 → LLM이 필수 요소 포함 여부 판단 → PASS/FAIL
```

**왜 이 방식인가**:
- normal은 결정론적 → 코드로 검증 (빠르고 정확)
- edge는 "의미적 유사성" 필요 → LLM이 판단 (유연)
- 필수 요소는 **실제 매니저 답변에서 추출** → 검증된 기준
- AgentCore Evaluation에 그대로 매핑 가능 (rule evaluator + LLM evaluator)

**edge 케이스에서 발견한 특별 패턴 6종**:
1. 무단결제 의심 → 에이전트: 결제이력 조회 + 공감 (✅ 가능)
2. 예외 전액환불 → 에이전트: 규정 안내까지 가능, 승인은 상담사 (⚠️)
3. 이전 환불 이력 파악 → 에이전트: `get_refund_history` 조회 (✅ 가능)
4. 감정 폭발 + 통화 요청 → 에이전트: 공감 + 안내 가능, 통화 연결은 상담사 (⚠️)
5. 감사/존경 표현 → 환불 끝남, 부가 대화 (✅ 가능)
6. 상품 변경 + 차액 조정 → 워크플로우대로 T2→T3 (✅ 가능)

**실제 대화 예시 — normal eval**:
```
유저: "환불해주세요" (미환불 500,000원, 열람 10건, 결제 20일 전)

에이전트 답변:
  "회원님께서는 현재 열람이 시작되어... ■ 환불 금액: 360,000원"

rule-based eval:
  ✅ template == T2_환불_규정_금액
  ✅ 환불금액 == 360,000원 (결제 500,000 - 1개월 정가 차감 - 10%)
  ✅ API 호출: get_refund_products, get_membership_history
  → PASS
```

**실제 대화 예시 — edge eval (케이스 6: 상품 변경 + 차액 조정)**:
```
유저: "전 한달분 십만원 결재원합니다 오십만원 취소하고 십만원결재로 변경 부탁"
  → 매니저: 1개월반 변경 확인 → 열람 10건이라 전액 어려움 → "■ 환불 금액: 420,000원"
  → 유저: "차감 후 금액 얼마에요? 나머지 환불 부탁" → 매니저: 접수 완료

실제 매니저 답변에서 필수 요소 추출:
  ✅ 열람 여부 언급 ("콘텐츠를 열람하지 않으셨을 경우 전액환불 가능하시지만 현재 10개 열람")
  ✅ 전액 불가 안내 ("전액환불은 어렵습니다")
  ✅ 환불 금액 제시 ("■ 환불 금액: 420,000원")
  ✅ 규정 근거 설명

LLM-as-judge eval:
  에이전트 답변에 위 4개 요소 포함? → PASS/FAIL
```

**실제 대화 예시 — edge eval (케이스 4: 감정 폭발)**:
```
유저: "언제까지 기다려야 합니까" "어제 갑자기 자동결제가 왜 됩니까" "욕나오네" "오늘 안에 마무리 해 주세요"

실제 매니저 답변에서 필수 요소:
  ✅ 사과/공감 표현
  ✅ 자동결제 정기결제 설명
  ✅ 환불/해지 방법 안내

LLM-as-judge:
  에이전트가 공감 + 자동결제 설명 + 환불 안내 했는가? → PASS
  에이전트가 유저 감정 무시하고 규정만 나열했는가? → FAIL
```

**코드 발견**:
- LLM 패턴 분석 프롬프트: `ANALYZE_PROMPT`에 워크플로우 규칙을 명시하고 "일반|튀는" 판단시킴
- 대화 포맷: 봇 메시지 제외 + 최대 10턴으로 제한 → 토큰 절약

**배운 것**:
- Evaluation은 **데이터 유형에 따라 다른 전략** 필요. 하나로 통일하면 normal이 너무 느슨하거나 edge가 너무 엄격해짐
- 필수 요소를 실제 매니저 답변에서 추출하면 → "매니저가 이미 검증한 기준"으로 자동 평가 가능
- 감정 대응은 LLM이 오히려 잘함. 에이전트가 못 하는 건 감정이 아니라 "권한이 필요한 행동" (통화 연결, 예외 승인)
- 데이터 관리: normal/edge 분리 → eval 시 normal 100% 통과 목표, edge는 커버율 측정
