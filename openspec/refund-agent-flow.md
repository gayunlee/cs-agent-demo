# 환불/해지 Agent 플로우 설계

> 2026-04-02 기준. 389건 실제 데이터 + 340건 매니저 응답 분석 기반.

## 핵심 발견

**유저 메시지만으로는 매니저 답변 유형을 예측할 수 없다.**
- 매니저는 **유저 정보를 조회한 후** 답변 유형을 결정
- "해지요청" → 조회해보니 이미 열람 → 환불 규정 안내 (T2)
- "환불해줘" → 조회해보니 첫 결제 직후 → 자동결제 설명 + 리텐션 (T4)

따라서 agent 플로우는:
```
1. 유저 메시지 수신
2. 정보 조회 (userId로 구독/결제/열람 이력)
3. 조회 결과 기반 → 답변 유형 결정
4. 템플릿 + 변수 채우기 → 초안 생성
```

## 턴별 분류

### A. 싱글턴 — 조회 없이 바로 답변

**적용 조건**: 유저 의도가 명확하고, 고정 템플릿으로 해결 가능

#### A-1. 구독해지 방법 안내 (T1)

```
유저: "해지하고 싶다" / "구독취소" / "탈퇴"
  │
  └─ 가입경로 확인 (앱/웹)
      ├─ 앱 → T1a (앱 해지 안내)
      └─ 웹 → T1b (웹 해지 안내)
```

- 도구: `get_subscriptions` (선택적, 앱/웹 판단용)
- 트리거: "해지", "구독취소", "탈퇴", "자동결제 끊"
- **주의**: "해지"라고 해도 실제로는 환불을 원하는 경우 있음 → 조회 후 분기

#### A-2. 본인확인 요청 (T6)

```
유저: (userId 매핑 불가 상태에서) 환불/해지 문의
  │
  └─ T6: "성함/휴대전화 번호 말씀 주시면 확인 도와드리겠습니다"
```

- 도구: 없음
- 트리거: userId가 없을 때 폴백

#### A-3. 카드변경 안내 (T8)

```
유저: "카드 변경" / "카드 분실" / "재발급"
  │
  └─ T8: 고정 템플릿 (웹페이지 카드변경 방법)
```

- 도구: 없음
- 트리거: "카드변경", "카드 분실", "카드 만료"

### B. 싱글턴 — 조회 후 바로 답변

#### B-1. 해지 확인 (T7)

```
유저: "해지됐나요?" / "처리됐는지 확인"
  │
  └─ [조회] get_subscriptions
      ├─ 구독 inactive → T7: "구독해지 잘 되어 있습니다"
      └─ 구독 active → T1: 해지 방법 안내
```

- 도구: `get_subscriptions`
- 트리거: "해지됐", "처리됐는지", "자동결제해지는 따로"

#### B-2. 환불 접수 완료 알림 (T3)

```
(환불 처리 후 → 상담사가 트리거)
  │
  └─ T3: "환불 접수 완료되었습니다" + 카드사 반영 안내
```

- 도구: 없음 (이미 처리된 후)
- 트리거: 환불 처리 완료 시점 (상담사 판단)

### C. 멀티턴 — 조회 후 판단 → 추가 질문 가능

#### C-1. 환불 규정 + 금액 안내 (T2)

```
유저: "환불해주세요" / "환불 가능?" / "남은 기간 환불"
  │
  └─ [조회] get_refund_products + get_membership_history
      │
      ├─ 7일 이내 + 미열람 → 전액 환불 가능 안내
      │   └─ "환불 진행 희망하시면 말씀 부탁드립니다"
      │
      ├─ 7일 이내 + 열람 → 부분 환불 안내
      │   └─ [calculate_refund] → "■ 환불 금액: {X}원"
      │   └─ "상기 환불금액 확인부탁드리며, 말씀주시면 접수 도와드리겠습니다"
      │
      ├─ 7일 경과 → 부분 환불 안내 (1개월 정가 차감 + 10%)
      │   └─ [calculate_refund] → 금액 제시
      │
      └─ 구독 여러 개 → 추가 질문 생성
          └─ "어떤 과정에 대한 환불을 희망하시나요?"
```

- 도구: `get_refund_products`, `get_membership_history`, `calculate_refund`
- **핵심**: 조회 결과에 따라 답변 분기가 달라짐
- 추가 질문이 필요한 경우: 복수 구독, 상품 특정 불가

#### C-2. 자동결제 설명 + 리텐션 (T4)

```
유저: "자동으로 결제됐다" / "왜 결제됐냐" / "결제된 줄 몰랐다"
  │
  └─ [조회] get_refund_products
      │
      └─ T4 템플릿:
         - "{마스터명} 마스터 과정 ... 구독형 스터디입니다"
         - "지난 {이전월} 과정에 이어 {현재월} 정기결제가 진행되었습니다"
         - "환불 혹은 해지 요청 주신다면 언제든 도와드릴 수 있습니다"
         - "이용하시면서 불편하셨던 점이 있으셨을까요?" ← 리텐션 질문
```

- 도구: `get_refund_products` (마스터명, 결제월 추출)
- **핵심**: 리텐션 시도. 환불/해지를 바로 처리하지 않고 이유를 물어봄
- 다음 턴: 유저가 환불 원하면 → C-1 (환불 규정) 분기

## 정보 조회 → 답변 유형 결정 매트릭스

| 조회 결과 | 유저 의도 | 답변 유형 |
|-----------|----------|-----------|
| 구독 없음 | 해지 | T6 (본인확인) 또는 "이미 해지됨" |
| 구독 active + 해지 문의 | 해지 | T1 (해지 방법) |
| 구독 active + 환불 문의 | 환불 | T2 (환불 규정 + 금액) |
| 구독 active + 자동결제 불만 | 환불/해지 | T4 (자동결제 설명 + 리텐션) |
| 구독 inactive + 해지 확인 | 확인 | T7 (해지 완료) |
| 최근 결제 + 미열람 | 환불 | T2 전액환불 가능 안내 |
| 최근 결제 + 열람 있음 | 환불 | T2 부분환불 금액 안내 |
| 복수 구독 | 환불/해지 | 추가 질문: "어떤 과정?" |
| userId 없음 | * | T6 (본인확인) |

## 구현 계획

### Phase 1: 싱글턴 처리 (T1, T6, T7, T8)
- 조회 없이 또는 구독상태만 조회해서 바로 답변
- 검증: 매니저 응답과 템플릿 일치 여부

### Phase 2: 멀티턴 처리 (T2, T4)
- 결제 + 열람 이력 조회 → 환불 계산 → 금액 포함 답변
- 검증: 환불 금액 정확성, 템플릿 변수 채우기

### Phase 3: T99 기타 처리
- 정형 템플릿에 안 맞는 케이스
- LLM 자유 생성 + RAG (과거 매니저 답변 참조)
- 검증: CS팀 직접 평가

### Phase 4: 추가 질문 생성
- 조회 결과 → 부족한 정보 감지 → 확인 질문 자동 생성
- "어떤 과정 환불 희망하시나요?" / "환불 진행 희망하실까요?"

## Eval 체크리스트

### Step 1: 템플릿 매칭
- [ ] 340건 각각에 올바른 답변 유형이 매칭되는가
- [ ] 의도 분류(유저 메시지 기반) vs 실제 분류(조회 결과 기반) 차이 분석
- [ ] T99_기타 208건의 세부 패턴 분석 → 추가 템플릿 필요 여부

### Step 2: 도구 호출
- [ ] 각 템플릿에 정의된 required_tools가 정확히 호출되는가
- [ ] API 응답 파싱이 올바른가 (amount가 string인 경우 등)
- [ ] 에러 핸들링 (API 실패, 빈 응답 등)

### Step 3: 변수 채우기
- [ ] {환불금액}, {마스터명} 등이 조회된 값으로 정확히 채워지는가
- [ ] 환불 계산 금액이 매니저 실제 제시 금액과 일치하는가
- [ ] 날짜 형식, 금액 포맷(쉼표) 등 디테일

### Step 4: 최종 답변 품질
- [ ] 완성된 답변이 실제 매니저 응답과 유사한가 (구조, 톤, 정보량)
- [ ] 불필요한 정보가 포함되지 않는가
- [ ] 고객에게 혼란을 줄 수 있는 표현이 없는가
