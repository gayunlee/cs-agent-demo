# CS AI 에이전트 MVP 기획안

> 작성일: 2026-03-31
> 기반: CS팀 미팅 (2026-03-31) + 채널톡 데이터 분석 (4,890건, 3개월)

---

## 1. 문제 정의

### 1-1. 워크플로우 봇 이탈 유저

현재 채널톡 워크플로우 봇은 버튼 메뉴(환불, 구독, 오류 등)를 제공하지만, **유저가 버튼만 누르고 실제 문의를 남기지 않는 경우**가 많다. 이 경우 매니저가 수동으로 "무엇을 도와드릴까요?"를 보내야 한다.

- 워크플로우 봇에서 "환불" 선택 → 환불 신청했다고 인식 → 이후 "왜 안 되냐"고 항의
- 이름만 입력하고 떠나는 유저도 상당 비중

**목표**: 봇 버튼 선택 후 추가 문의가 없는 유저에게 유형별 추가 질문을 자동 전송, 의도를 파악하여 적절한 응답 생성

### 1-2. 환불 안내 자동화

전체 상담의 33.7%(1,647건/3개월)가 환불 관련. 현재:

- CS팀이 결제 금액, 결제 주기, 열람 여부를 **수기 조회 + 수기 계산**
- 환불 정책은 **담당자 머릿속**에만 존재 (문서화 안 됨)
- 어스캠퍼스는 구글시트에 별도 환불 이력 제출 필요
- 정책 변경 가능성 있어 하드코딩 불가

**목표**: 유저 정보 자동 조회 + 동적 환불 규정 테이블 기반 자동 계산 + 안내 답변 생성 (처리는 매니저가)

## 2. MVP 스코프 (확정: 2026-03-31)

### Phase 1 — MVP 데모 (2026-04-01)

| # | 기능 | 설명 | 우선순위 |
|---|------|------|:--------:|
| 1 | 채널톡 webhook 수신 | 문의 생성 이벤트 수신 엔드포인트 | P0 |
| 2 | 유저 식별 + 정보 조회 | webhook → memberId/전화번호 → 관리자센터 API로 유저 정보 자동 조회 | P0 |

| 4 | 워크플로우 봇 이탈 처리 | 봇 버튼만 누른 유저 패턴 감지 → 유형별 추가 질문 자동 전송 | P0 |
| 5 | 답변 생성 파이프라인 | 의도 분류 → 템플릿 매칭 → 변수 자동 채움 → 매니저에게 초안 제공 | P0 |

## 3. 아키텍처

### 3-0. 전체 플로우

```
채널톡 webhook (문의 생성 이벤트)
  │
  ├─ [수신] POST /webhook/channeltalk
  │     └─ 이벤트 파싱: chatId, memberId, 메시지 내용
  │
  ├─ [유저 식별] memberId or 전화번호 → 관리자센터 userId
  │     ├─ GET /v3/users?phoneNumber=xxx → userId 획득
  │     └─ (프로덕션: 채널톡 초기화 시 userId 주입 → 직접 매칭)
  │
  ├─ [유저 정보 조회] 관리자센터 API (7개 핵심 엔드포인트)
  │     ├─ GET /v1/users/{id}                        → 기본정보
  │     ├─ GET /users/{id}/my-products                → 보유 상품, 구독 상태
  │     ├─ GET /cs/refund-user/{userId}/products      → 결제 정보 (금액, 카드, 주기)
  │     ├─ GET /cs/refund-user/{userId}/shop-product  → 샵 결제 내역
  │     ├─ GET /users/{id}/contents                   → 콘텐츠 열람 여부
  │     ├─ GET /v1/users/{id}/membership-history      → 구독 이력
  │     └─ GET /my-products/{id}/transaction-round-history → 빌링 사이클
  │
  ├─ [감지] 워크플로우 봇 이탈 패턴?
  │     ├─ Type A (봇 메뉴만 클릭) → 유형별 추가 질문 전송
  │     ├─ Type B (trivial 텍스트) → "무엇을 도와드릴까요?"
  │     ├─ Type C (환불 버튼 + 상세 미입력) → 환불 관련 안내 질문
  │     └─ 실제 문의 있음 → 아래 플로우
  │
  ├─ [분류] 의도 분류 (4분류 + 환불 세부)
  │
  ├─ [환불인 경우] 동적 환불 규정 테이블 적용
  │     ├─ 조건 매칭: 경과일, 열람 여부, 구독/단건, 결제 주기
  │     ├─ 금액 계산: 검증된 공식 적용
  │     │     차감금 = 1개월_정가 × (경과_비율)
  │     │     잔여금 = 결제총액 - 차감금
  │     │     수수료 = 잔여금 × 10%
  │     │     환불액 = 잔여금 - 수수료
  │     └─ 결과: 환불 가능 여부 + 금액 + 근거
  │
  ├─ [생성] 답변 초안 생성
  │     ├─ 16종 템플릿 매칭 → {환불금액} 자동 채움
  │     └─ or LLM 생성 (RAG 참고 답변 + 조회 정보)
  │
  └─ [전달] 매니저에게 답변 초안 + 조회 결과 제공
        └─ 매니저가 확인/수정 후 전송
```

### 3-1. 유저 식별 흐름

```
[데모] 전화번호 → GET /v3/users → userId → 나머지 API 호출
[프로덕션] 채널톡 초기화 시 userId 주입 (프론트엔드 배포 필요)
  → 채널톡 memberId = 관리자센터 userId → 직접 매칭
```

### 3-2. 관리자센터 API 인증

```
[데모] 브라우저 개발자도구에서 인증 토큰 복사 → .env ADMIN_API_TOKEN
[프로덕션] 내부 서비스용 API key 발급 → CS 에이전트 전용 서비스 계정
```

### 3-3. 동적 환불 규정 테이블

데이터 탐색에서 검증된 5개 규칙 기반 (→ `openspec/data-exploration-results.md` 참조):

| ID | 조건 | 환불 유형 | 신뢰도 |
|----|------|-----------|:------:|
| EP-001 | 7일 이내 + 콘텐츠 미열람 | 전액환불 | high |
| EP-002 | 개시 + 1/3 기간 경과 전 | 정가 1/3 차감 + 잔여 10% 수수료 | high |
| EP-003 | 1/3 ~ 1/2 기간 경과 | 정가 1/2 차감 + 10% 수수료 (추정) | medium |
| EP-004 | 1/2 기간 경과 후 | 환불불가 (추정) | low |
| EP-005 | N개월 경과 | N개월 정가 차감 + 잔여 10% 수수료 | high |

MVP에서는 신뢰도 high인 EP-001, EP-002, EP-005 우선 구현. 나머지는 CS팀 확인 후 추가.

규정 테이블은 JSON config로 관리 → 정책 변경 시 코드 수정 불필요.

## 4. 테스트 / 검증 방법

### 4-1. 워크플로우 봇 이탈 패턴

| 검증 항목 | 방법 |
|-----------|------|
| 패턴 정확도 | 채널톡 데이터에서 실제 이탈 케이스 추출 → 에이전트가 올바른 추가 질문을 보내는지 확인 |
| 유형 분류 정확도 | 이탈 후 추가 질문에 응답한 케이스로 의도 분류 정확도 측정 |
| 커버리지 | 전체 이탈 유저 중 에이전트가 처리 가능한 비율 |

### 4-2. 환불 안내

| 검증 항목 | 방법 |
|-----------|------|
| 환불 금액 계산 정확도 | CS팀이 실제 수기 계산한 결과 vs 에이전트 계산 결과 비교 (과거 환불 케이스) |
| 규정 매칭 정확도 | 경계 조건 테스트 (7일 째, 열람 1회 vs 0회, 구독 vs 단건 등) |
| 답변 품질 | CS팀에게 생성된 답변 보여주고 "이대로 보낼 수 있는지" 평가 |

### 4-3. 전체 파이프라인

| 검증 항목 | 방법 |
|-----------|------|
| E2E 시뮬레이션 | 실제 채널톡 대화 데이터로 전체 플로우 실행, 생성된 답변을 CS팀 검수 |
| 오분류 안전장치 | 환불 금액이 실제와 다를 경우 매니저에게 경고 표시 |
| fallback | 에이전트가 확신 없을 때 매니저 에스컬레이션 정상 동작 확인 |

## 5. 태스크 구현 계획

### Task 0: 데이터 탐색 및 패턴 파악

#### 0-1. 워크플로우 봇 이탈 패턴 탐색

**목적**: 채널톡 데이터에서 "봇 버튼만 누르고 실제 문의를 안 남긴" 대화 패턴을 찾아 유형화

**탐색 지침**:
1. `letter-post-weekly-report/data/channel_io/` 및 BigQuery `channel_io.messages` 테이블에서 데이터 확인
2. 봇 메시지(워크플로우) 이후 유저 메시지가 없거나, 이름/인사만 남긴 대화 필터링
   - `personType = "bot"` 메시지 이후 `personType = "user"` 메시지 부재 or 짧은 메시지(10자 이내)
   - 워크플로우 버튼 클릭 이벤트 (`pluginMessage` 등) 이후 패턴
3. 이탈 유형 분류:
   - 버튼만 클릭하고 떠남
   - 이름만 입력하고 떠남
   - "환불" 버튼 눌렀지만 환불 상세 미입력
   - 기타
4. 각 유형별 건수, 비중, 대표 예시 정리
5. 유형별로 어떤 추가 질문이 적절한지 제안

**출력**: 이탈 패턴 유형 목록 + 건수 + 예시 + 추가 질문 제안

#### 0-2. 답변 템플릿 분석

**목적**: 매니저가 실제 사용하는 답변 템플릿과 수기 변경 값 파악

**탐색 지침**:
1. `letter-post-weekly-report/data/channel_io/` 에서 매니저 메시지 추출
2. 반복 사용되는 정형 답변 패턴 추출 (유사도 클러스터링 or 빈도 분석)
3. 기존 분석 결과와 대조 (구독해지 33회, 카드변경 25회, 시스템환불 21회 등 12종)
4. 각 템플릿에서 **수기 변경이 필요한 값** 식별:
   - `{만료일}`, `{환불금액}`, `{마스터명}`, `{결제일}`, `{카드정보}` 등
5. 해당 값들이 관리자센터 어떤 API에서 조회 가능한지 매핑

**출력**: 템플릿 목록 + 변수(placeholder) 목록 + 필요 API 매핑

#### 0-3. 관리자센터 API 탐색

**목적**: 유저 정보 자동 조회에 필요한 API 엔드포인트 파악

**탐색 지침**:
1. 관리자센터 API 문서 또는 실제 엔드포인트 확인
2. 필요한 정보별 API 매핑:
   - 계정 정보 (이름, 연락처, 가입방법) → 어떤 API?
   - 구독 상태 (마스터, 상품, 등급, 상태) → 어떤 API?
   - 결제 이력 (금액, 일자, 카드, 결제주기) → 어떤 API?
   - 콘텐츠 열람 이력 → 어떤 API?
3. 채널톡 유저 식별 → 관리자센터 유저 매칭 방법:
   - 채널톡 초기화 시 `userId` 주입 여부 확인
   - 채널톡 `memberId` ↔ 관리자센터 계정 ID 매칭 가능 여부
   - 연락처(전화번호) 기반 매칭 가능 여부
4. API 인증 방식, rate limit, 응답 형태 정리

**출력**: API 엔드포인트 목록 + 요청/응답 스펙 + 유저 매칭 방법

#### 0-4. 환불 계산 로직 설계

**목적**: 동적 환불 규정 테이블 구조 확정 + 계산 로직 구현 준비

**탐색 지침**:
1. 채널톡 데이터에서 실제 환불 처리된 대화 추출 (1,647건/3개월)
2. 매니저가 안내한 환불 금액과 조건 패턴 분석:
   - 전액 환불 조건
   - 부분 환불 계산식 (일할 계산, 이용일수 차감 등)
   - 환불 거절 조건
   - 구독(정기결제) vs 단건 구매 차이
3. 어스캠퍼스 vs 어스플러스 환불 처리 차이 파악
4. 결과로 규정 테이블 초안 작성

**출력**: 환불 규정 테이블 초안 (JSON) + 계산식 + 플랫폼별 차이
